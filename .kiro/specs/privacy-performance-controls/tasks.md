# Implementation Plan

- [ ] 1. Foundation: controls settings, storage, and policy types
- [ ] 1.1 Define privacy, provider, resource, status, storage, and error contracts
  - Establish strongly typed settings, work requests, decisions, blocked reasons, resource budgets, status snapshots, storage usage, cleanup results, and policy impact plans.
  - Define the canonical `ProviderMode = "localOnly" | "remoteAllowed" | "hybrid"` enum and document that semantic and vision consume it only through thin adapters.
  - Preserve the Local File Indexer distinction between canonical `fileType` and normalized `extension`, including distinct exclusion rule kinds for each.
  - The observable result is a shared controls type module that all pipeline and shell integration work can import without using unsafe types.
  - _Requirements: 1.1, 1.2, 1.3, 2.1, 2.4, 2.6, 3.1, 3.2, 3.3, 3.4, 4.4, 6.4, 7.1, 8.5_
- [ ] 1.2 Add persisted controls settings and migration support
  - Store indexed locations, exclusions, provider consent, resource limits, battery settings, pause state, and policy version.
  - Preserve prior settings on failed writes and increment policy version after accepted changes.
  - The observable result is controls settings surviving restart and returning a privacy-preserving default configuration.
  - _Requirements: 1.1, 2.1, 2.2, 4.1, 4.4, 5.1, 5.4, 8.2, 8.4_
  - _Touches: storage migrations_

- [ ] 2. Core policy evaluation
- [ ] 2.1 (P) Implement indexed scope and exclusion evaluation
  - Evaluate approved roots, nested excluded folders, canonical file type exclusions, normalized extension exclusions, and restrictive-precedence behavior.
  - Return user-presentable denied reasons and policy impact data for settings changes.
  - The observable result is file work requests being allowed or denied according to root and exclusion settings.
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 3.1_
  - _Boundary: PrivacyPolicy_
- [ ] 2.2 (P) Implement provider mode and consent evaluation
  - Enforce local-only defaults and require explicit consent for remote processing by provider and modality.
  - Produce regeneration, invalidation, or current-state impact plans when provider policy changes.
  - The observable result is remote content requests denied until consent exists and policy allows the modality.
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 2.5, 2.6, 3.2, 3.3, 3.4, 3.5_
  - _Boundary: PrivacyPolicy_
- [ ] 2.3 (P) Implement resource, pause, throttle, and battery policy
  - Evaluate pause state, concurrency, CPU, memory, disk, battery-aware mode, and user battery override.
  - Distinguish temporary defer reasons from hard deny reasons.
  - The observable result is background work receiving paused, throttled, waiting-for-power, or allowed decisions under configured conditions.
  - _Requirements: 4.1, 4.2, 4.3, 4.4, 4.5, 5.1, 5.2, 5.3, 5.4, 5.5_
  - _Boundary: ResourcePolicy_
- [ ] 2.4 Compose the shared policy decision service
  - Combine privacy and resource evaluators into one allow, deny, or defer decision surface.
  - Fail closed for private content processing when policy cannot be evaluated.
  - The observable result is one service used for file discovery, extraction, visual processing, content embedding, and query embedding decisions.
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 3.5_

- [ ] 3. Pipeline enforcement integrations
- [ ] 3.1 (P) Integrate policy decisions into Local File Indexer eligibility
  - Consult controls policy before new `PipelineWorkItem` records are queued for files under approved roots.
  - Pass both canonical `fileType` and normalized `extension` in `PolicyWorkRequest` values when known.
  - Re-evaluate known files after scope or exclusion settings change.
  - The observable result is excluded or policy-denied files recorded with explainable status instead of being processed.
  - _Requirements: 1.2, 1.3, 1.5, 3.1, 6.2, 6.3, 6.4_
  - _Boundary: Local File Indexer Policy Hook_
  - _Depends: 2.4_
  - _Touches: src/indexer/jobService.ts, storage migrations_
- [ ] 3.2 (P) Integrate policy decisions into Content Extraction Pipeline
  - Consult controls policy before reading file contents or retrying extraction work.
  - Defer extraction when paused, throttled, or waiting for power.
  - The observable result is extraction work starting only after an allow decision and preserving retryable deferred state otherwise.
  - _Requirements: 2.3, 3.2, 4.1, 4.3, 4.5, 5.3, 6.2_
  - _Boundary: Content Extraction Policy Hook_
  - _Depends: 2.4_
  - _Touches: src/extraction/extractionWorker.ts_
- [ ] 3.3 (P) Integrate policy decisions into Vision OCR Pipeline
  - Consult controls policy through `VisionPolicyAdapter` before local or remote OCR, captioning, and tagging.
  - Enforce remote consent and battery-aware deferral for expensive visual work.
  - The observable result is visual work blocked, deferred, or allowed with status reasons matching controls policy.
  - _Requirements: 2.1, 2.2, 2.3, 3.3, 4.5, 5.1, 5.3, 6.2_
  - _Boundary: Vision Policy Hook_
  - _Depends: 2.4_
  - _Touches: src/vision/visionWorker.ts_
- [ ] 3.4 (P) Integrate policy decisions into Semantic Vector Search
  - Delegate content and query embedding provider policy to controls decisions through `EmbeddingPolicyAdapter`.
  - Invalidate or regenerate embeddings when provider policy impact plans require it.
  - The observable result is embeddings and query provider calls blocked unless provider mode and consent allow them.
  - _Requirements: 2.1, 2.3, 2.5, 3.4, 5.5, 6.2_
  - _Boundary: Semantic Policy Hook_
  - _Depends: 2.4_
  - _Touches: src/semantic-search/providers/embeddingPolicyAdapter.ts, src/semantic-search/intake/semanticContentIntake.ts_

- [ ] 4. Settings status, storage, and cleanup services
- [ ] 4.1 (P) Build normalized controls status aggregation
  - Combine indexer, extraction, visual, semantic, and policy state into current, pending, failed, excluded, policy-blocked, throttled, and removed counts.
  - Convert domain-specific pipeline details into user-presentable blocked reasons.
  - The observable result is a shell-safe status snapshot that explains incomplete search coverage without raw content.
  - _Requirements: 4.2, 4.5, 5.3, 6.1, 6.2, 6.3, 6.4, 6.5, 8.5_
  - _Boundary: ControlsStatusAggregator_
- [ ] 4.2 (P) Build storage visibility and cleanup coordination
  - Aggregate approximate storage usage for metadata, extracted content, visual payloads, embeddings, and vectors.
  - Coordinate per-location and full-index cleanup through upstream cleanup ports.
  - The observable result is storage usage visible by category and cleanup making selected local index data unavailable for search until rebuilt.
  - _Requirements: 7.1, 7.2, 7.3, 7.4, 7.5_
  - _Boundary: StorageVisibilityService, CleanupCoordinator_
- [ ] 4.3 Build controls view model service
  - Compose settings, consent previews, status snapshots, storage usage, cleanup states, and errors for the shell.
  - Preserve prior settings when updates fail and require explicit consent before remote mode is applied.
  - The observable result is one shell-facing view model with accepted changes, rejected changes, and consent-required previews.
  - _Requirements: 2.2, 2.4, 6.5, 8.1, 8.2, 8.3, 8.4, 8.5_

- [ ] 5. Desktop settings UI integration
- [ ] 5.1 Add privacy and performance settings panels to the shell
  - Display indexed locations, exclusions, provider mode, pause/resume, resource limits, battery behavior, indexing status, and storage controls.
  - Keep controls text high-level and avoid parser, OCR, embedding, vector, or raw content details.
  - The observable result is a settings surface where users can review and change all controls from the desktop app.
  - _Requirements: 1.1, 1.2, 1.3, 4.1, 4.3, 4.4, 5.4, 6.1, 7.1, 8.1, 8.5_
  - _Touches: src/app/App.tsx_
- [ ] 5.2 Implement remote consent and settings error flows
  - Show explicit consent before remote provider mode is applied.
  - Display accepted state after successful updates and preserve prior state with actionable errors on failures.
  - The observable result is remote processing remaining disabled until the consent step completes successfully.
  - _Requirements: 2.2, 2.3, 8.2, 8.3, 8.4, 8.5_

- [ ] 6. Validation and regression coverage
- [ ] 6.1 Add unit tests for privacy and resource policy decisions
  - Cover exclusions, provider consent, fail-closed behavior, pause, throttling, battery deferral, and policy-change impact planning.
  - The observable result is policy tests proving every decision reason used by pipeline hooks.
  - _Requirements: 1.2, 1.3, 1.4, 1.5, 2.1, 2.2, 2.3, 2.5, 3.5, 4.1, 4.5, 5.1, 5.4_
- [ ] 6.2 Add integration tests for pipeline hooks and status aggregation
  - Verify indexer, extraction, vision, and semantic hooks all honor allow, deny, and defer decisions.
  - Verify aggregated status and blocked reasons match upstream and policy states.
  - The observable result is cross-pipeline tests failing if any processor bypasses policy.
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 6.1, 6.2, 6.3, 6.4, 6.5_
- [ ] 6.3 Add UI and cleanup workflow tests
  - Cover settings display, remote consent, pause/resume, storage usage, cleanup success, and cleanup failure.
  - Verify raw content, OCR text, captions, embeddings, and vectors do not appear in controls.
  - The observable result is end-to-end coverage for the critical user-facing controls and privacy expectations.
  - _Requirements: 7.1, 7.2, 7.3, 7.4, 7.5, 8.1, 8.2, 8.3, 8.4, 8.5_
