# Implementation Plan

- [ ] 1. Foundation: semantic storage, configuration, and contracts
- [ ] 1.1 Define semantic search contracts, semantic intake, and provider policy adapter inputs
  - Establish typed contracts for `SemanticContentInput`, semantic chunks, embedding records, vector search matches, freshness states, query results, status snapshots, and normalized semantic errors.
  - Include provider model metadata and the adapter inputs needed to ask Privacy Performance Controls whether content or query text may be embedded.
  - Use shared `ProviderMode = "localOnly" | "remoteAllowed" | "hybrid"` names from Privacy Performance Controls.
  - The completed contracts compile without unsafe public types and can be imported by semantic services, provider adapters, and tests.
  - _Requirements: 1.4, 3.1, 3.2, 3.4, 4.5, 5.1, 7.1, 8.5_
- [ ] 1.2 Create local semantic persistence and vector index foundations
  - Add semantic schema support for chunks, embedding records, freshness state, canonical `sourceVersion`, retry diagnostics, and current pointers.
  - Add a local vector index adapter boundary with vector upsert, invalidation, metadata-filtered search, and stats behavior.
  - The completed foundation can persist semantic records and recover vector index metadata after a simulated restart.
  - _Requirements: 4.1, 4.2, 4.3, 4.4, 8.3_
  - _Touches: storage migrations_

- [ ] 2. Core indexing pipeline
- [ ] 2.1 (P) Build provider-neutral semantic content intake
  - Consume current or limited extracted payloads with usable text from the extraction reader.
  - Consume current or limited visual payloads with usable OCR, caption, or tag text from the visual reader.
  - Preserve file ID, source kind, upstream artifact ID, canonical `sourceVersion`, file path, file type, size, and modified timestamp in semantic work records.
  - Skip unsupported, empty, failed, pending, stale, policy-blocked, and removed upstream states without creating embedding work.
  - The completed intake produces semantic work only through `SemanticContentInput`, so vision does not patch semantic internals later.
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5_
  - _Boundary: SemanticContentIntake_
  - _Touches: src/semantic-search/intake/semanticContentIntake.ts_
  - _Depends: 1.1, 1.2_
- [ ] 2.2 (P) Build semantic chunk preparation
  - Split extracted text into ordered chunks using extraction hints where available and fallback boundaries where needed.
  - Preserve offsets, labels, ordinals, canonical `sourceVersion`, source kind, upstream artifact ID, file ID, and limited-content markers.
  - Generate deterministic chunk identifiers for the same canonical `sourceVersion`, source kind, upstream artifact ID, and boundaries.
  - The completed chunker returns traceable chunks for normal, hinted, oversized, and limited payloads.
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 2.5_
  - _Boundary: ChunkingService_
  - _Depends: 1.1_
- [ ] 2.3 (P) Build embedding policy adapter and provider service
  - Delegate local-first, remote-allowed, and hybrid provider decisions to Privacy Performance Controls before embedding content or queries.
  - Request embeddings through a provider-neutral interface and capture provider model metadata with returned vectors.
  - Preserve existing current embeddings when providers are unavailable and record retryable failure status.
  - The completed service can embed chunk batches and queries through a test provider while policy denials and provider failures are observable.
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 5.1, 5.4, 8.3_
  - _Boundary: EmbeddingService, EmbeddingPolicyAdapter_
  - _Depends: 1.1_
- [ ] 2.4 Implement semantic freshness transitions
  - Mark affected chunks and embeddings stale when semantic intake changes or Privacy Performance Controls invalidates prior embeddings.
  - Mark current canonical `sourceVersion` values current only after chunk records, embedding metadata, and vectors are successfully refreshed.
  - Mark deleted or removed file vectors unavailable so they cannot appear in current results.
  - The completed freshness behavior prevents stale, removed, and policy-invalidated vectors from being treated as current.
  - _Requirements: 3.5, 4.3, 4.4, 5.5, 8.1, 8.2_
  - _Depends: 2.1, 2.2, 2.3_
- [ ] 2.5 Integrate the background semantic index worker
  - Orchestrate semantic content intake, chunk preparation, embedding, vector upsert, freshness updates, and retryable failure recording.
  - Process work in bounded batches without blocking shell search submission.
  - The completed worker refreshes a ready semantic content input into current searchable vectors and reports failures without stopping subsequent inputs.
  - _Requirements: 1.1, 1.2, 2.1, 3.3, 4.1, 8.2, 8.3, 8.4_
  - _Depends: 2.1, 2.2, 2.3, 2.4_

- [ ] 3. Query retrieval and result generation
- [ ] 3.1 (P) Build semantic query retrieval
  - Validate non-empty natural language queries and create query embeddings under current provider policy.
  - Retrieve top matching chunks from the local vector index using current-state and provider-compatible filters.
  - Return an empty not-ready state when no current vectors exist and a provider-unavailable error when query embedding fails.
  - The completed query service never returns stale, removed, failed, or policy-invalidated vector matches.
  - _Requirements: 5.1, 5.2, 5.3, 5.4, 5.5_
  - _Boundary: SemanticQueryService_
  - _Depends: 1.2, 2.3, 2.4_
- [ ] 3.2 (P) Build file-level ranking and grouping
  - Group multiple chunk matches from the same file into one file-level candidate.
  - Combine semantic similarity with available file metadata signals and limited-content metadata.
  - Apply deterministic tie-breaking for equivalent relevance.
  - The completed ranking service returns stable ranked file candidates for repeated identical inputs.
  - _Requirements: 6.1, 6.2, 6.3, 6.4, 6.5_
  - _Boundary: RankingService_
  - _Depends: 1.1, 1.2_
- [ ] 3.3 (P) Build match context generation
  - Produce snippet or explanation match context from the best current chunk when safe context is available.
  - Preserve chunk labels and offsets where useful for human-readable context.
  - Omit match text rather than fabricating context when no safe snippet can be produced.
  - The completed context builder returns shell-compatible match context for explainable matches and no context for unsafe matches.
  - _Requirements: 2.4, 7.2, 7.4_
  - _Boundary: SnippetBuilder_
  - _Depends: 2.2_
- [ ] 3.4 Implement the shell-compatible semantic search provider
  - Adapt semantic query results into the Desktop Search Shell `SearchResult` contract.
  - Include file path, display name, file type, rank, action eligibility, optional match context, and bounded `availabilityHint` metadata for partial or unavailable indexing.
  - Normalize provider failures into shell-compatible errors and semantic not-ready states into empty `SearchResponse` values with query-level readiness metadata.
  - The completed provider can replace the placeholder provider without requiring desktop UI component changes.
  - _Requirements: 6.5, 7.1, 7.2, 7.3, 7.4, 7.5_
  - _Depends: 3.1, 3.2, 3.3_
  - _Touches: src/search/SearchProvider.ts_

- [ ] 4. Status, recovery, and diagnostics
- [ ] 4.1 Build semantic status snapshots
  - Aggregate pending, embedding, current, stale, failed, removed, and policy-invalidated counts.
  - Expose search readiness and degraded status without raw file contents, vectors, provider secrets, or storage internals.
  - The completed status service reports readiness and diagnostic counts that the shell or diagnostics can consume safely.
  - _Requirements: 4.5, 8.3, 8.5_
  - _Depends: 1.2, 2.4_
- [ ] 4.2 Build startup recovery and reconciliation
  - Recover persisted chunks, embeddings, freshness states, retry diagnostics, and vector index availability after restart.
  - Reconcile mismatches between semantic repository state and vector index availability.
  - The completed recovery path marks mismatches stale or degraded without requiring a full manual rebuild.
  - _Requirements: 4.2, 4.4, 8.1, 8.2, 8.3_
  - _Depends: 1.2, 2.4, 4.1_

- [ ] 5. Validation and integration tests
- [ ] 5.1 Add chunking, policy, and embedding unit coverage
  - Verify hinted chunking, oversized text splitting, deterministic chunk IDs, offsets, limited payload behavior, and provider-policy adapter delegation.
  - Verify provider unavailable, policy denied, and vector dimension mismatch errors preserve retryable or blocking status as designed.
  - The completed test suite fails on traceability loss, remote-by-default embedding, and unsafe provider behavior.
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 2.5, 3.1, 3.2, 3.3, 3.4_
  - _Depends: 2.2, 2.3_
- [ ] 5.2 Add indexing pipeline and freshness integration coverage
  - Verify extracted and visual semantic inputs become current chunk, embedding, and vector records through the same intake contract.
  - Verify unsupported, empty, failed, pending, removed, deleted, stale, and policy-invalidated states are excluded from current search.
  - Verify restart recovery preserves current records and marks repository/vector mismatches degraded or stale.
  - The completed tests demonstrate end-to-end semantic freshness behavior from extracted payload intake through vector availability.
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 4.1, 4.2, 4.3, 4.4, 8.1, 8.2, 8.3, 8.4_
  - _Depends: 2.5, 4.2_
- [ ] 5.3 Add query, ranking, and shell provider contract coverage
  - Verify natural-language query embedding, vector retrieval filters, not-ready empty results, and provider-unavailable errors.
  - Verify grouped file-level ranking, metadata signals, deterministic tie-breaking, match context, and no fabricated snippets.
  - Verify returned results satisfy the Desktop Search Shell contract, use bounded `availabilityHint` for partial indexing metadata, and do not expose semantic internals.
  - The completed tests prove semantic search can serve shell-compatible ranked results for current indexed content.
  - _Requirements: 5.1, 5.2, 5.3, 5.4, 5.5, 6.1, 6.2, 6.3, 6.4, 6.5, 7.1, 7.2, 7.3, 7.4, 7.5_
  - _Depends: 3.4_
