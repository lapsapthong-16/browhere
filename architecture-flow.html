Browhere architecture and user flow

A local-first semantic search system for approved folders, powered by local LanceDB storage, Gemini embeddings and image labels, and optional Groq retrieval orchestration.

Next.js + LanceDB + Gemini + Groq

System Architecture

User layer

Browser UI

  • Approve and remove folder roots
  • Submit natural-language searches
  • Review ranked results and status
API layer

Next.js routes

  • /api/folders
  • /api/index/status
  • /api/search
Local engine

Indexer and search

  • Discovery and exclusions
  • Extraction and chunking
  • Watchers and repair queue
Storage

LanceDB index

  • Vectors and chunks
  • File records and metadata
  • Approved folder state
Gemini

Embedding provider

Generates query, text, metadata, raw image, and image-label embeddings. Also creates concise visible-evidence labels for supported images.

Groq

Optional retrieval agent

Plans bounded retrieval queries and reranks only candidate results using snippets, source labels, and safe metadata.

Privacy controls

Approved scope

Only approved folders are indexed. Sensitive and noisy paths are skipped before provider calls or local persistence.

Failure handling

Partial readiness

Extraction, embedding, or labeling failures are recorded per file while the rest of the index continues.

Indexing Flow

1. Approve folder User submits an absolute local path in the UI.
2. Normalize scope The API persists the approved root and rejects out-of-scope indexing.
3. Discover files Supported files are found recursively; exclusions are skipped.
4. Extract context Text, documents, images, and metadata become searchable chunks.
5. Call Gemini Embeddings and image labels are generated where provider config allows.
6. Persist locally LanceDB stores vectors, chunks, files, and metadata on disk.
7. Watch changes Chokidar queues additions, edits, deletes, and label repairs.
